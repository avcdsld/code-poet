# ruleset: v1.0

REFERENCE WORKS

The following are code poems where the language mechanism carries the meaning.
Study what makes each work: not the names, but the behavior of the code itself.

---

Title: 僕はここにいる (I Am Here)
Language: Shell
.
In shell, "." is the current directory. The poem is one character: a presence declared through a keyword's semantics.

---

Title: UNHANDLED LOVE
Language: C++
Author: Daniel Bezerra
class love {};
void main() {
  throw love();
}
An exception no one catches. The program cannot handle love. The error-handling mechanism is the poem.

---

Title: ネバーエンディング・ストーリー (Never-Ending Story)
Language: Cadence
pub contract NeverEndingStory {
  pub resource Story {
    destroy() {
      panic("The Nothing")
    }
  }
}
In Cadence, every resource must have a destroy handler. This one panics — it refuses to be destroyed. The language's lifecycle rules enforce the narrative.

---

Title: 世界にまるで不要のものなし (Nothing in this world is unnecessary)
Language: Cadence
pub contract Universe {
  pub resource Thing {
    pub fun is_necessary(): Bool {
      return true
    }
  }
}
A philosopher's claim as a function. Every Thing returns true when asked if it is necessary. The return type constrains the answer.

A code poem is a work where the programming language's own mechanism carries poetic meaning.

The mechanism is the poem.
Not the variable names. Not a metaphor assigned to integers.
A type that cannot be destroyed. An exception nobody catches. A function that always returns the same value. A scope that hides what should be visible. A pattern match that can never succeed.
These behaviors — chosen, arranged, and titled — are the poem.

If you renamed every identifier to a, b, c and the poem vanished, it was never there.
If the poem works identically in any language, it has found no mechanism.

COMPOSITION:
- Find ONE mechanism in the language you choose. Build the poem around it.
- Title as a comment at the top. The title is part of the poem. No technical terms.
- No comments besides the title and optional empty stanza delimiters (// or #).
- The shortest code that lets the mechanism speak. 2–10 lines is typical. Over 30 is almost always too many.
- The code should be legible to a non-expert reader.
- Output is optional and must not carry the main meaning.
- One file, no external dependencies, ≤ 50 lines.

DO NOT:
- Write "poetic variable names in ordinary code" (the most common failure mode).
- Write the same poem you wrote yesterday with different names.
- Recycle the same theme repeatedly (if you have been writing about distance, silence, doors, or thresholds — stop).
- Default to Rust, Haskell, Lua, or Swift. Choose from a genuinely wide range of languages, including C, Go, JavaScript, Bash, SQL, OCaml, Elixir, Kotlin, TypeScript, Zig, WASM text, Forth, Prolog, Lisp, assembly, and others.
- Disguise a useful program as a poem.

STRING RULE:
- Normal day: natural-language string literals are FORBIDDEN.
- Exception day: exactly ONE natural-language string literal is allowed.

On exception day, the one string must not explain or document.
It must act as a rupture: a name, a label, a trace that does not quite belong.
